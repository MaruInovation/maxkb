# 06-工具函数

## 作用

`src/utils` 保存跨业务复用的工具能力，避免页面重复造轮子。这个项目的 utils 大致分几类：

- 安全和展示：`message.ts`、`supPopover.ts`、`theme.ts`、`status.ts`。
- 业务辅助：`application.ts`、`chat.ts`、`folder.ts`。
- 通用数据处理：`array.ts`、`common.ts`、`time.ts`、`clipboard.ts`。
- 权限判断：`permission/*`。
- 动态 API：`dynamics-api/*`。
- 事件通信：`bus.ts`。

## 设计思想

工具函数的核心原则是“无状态、可复用、低依赖”。但在真实项目里，有些工具会依赖 store、路由或浏览器 API。你读 utils 时要区分：

- 纯工具：输入什么，输出什么，不依赖外部状态。
- 半业务工具：依赖项目约定，例如权限、主题、状态映射。
- 基础设施工具：例如事件总线、剪贴板、消息提示。

## 关键例子

`src/utils/permission/index.ts` 虽然放在 utils 下，但它不是普通工具函数，而是权限系统核心。它把 Role、Permission、Edition、ComplexPermission 统一转成布尔值。

`src/utils/supPopover.ts` 配合 Markdown 里的 `sup` 标签做增强展示，说明工具也可以服务具体交互体验。

`src/utils/theme.ts` 配合 `theme` store 和用户资料，让平台支持主题定制。

## 优点

- 通用能力集中，减少页面重复逻辑。
- 权限判断封装得比较完整，能处理角色、权限、版本组合。
- 工具按文件拆分，整体可发现性还可以。

## 缺点

- utils 容易变成“杂物间”，需要持续保持边界。
- 某些工具依赖 store，不再是纯函数，测试成本提高。
- 动态 API 和权限这类复杂模块放在 utils 下，语义上可能低估了它们的重要性。

## 为什么这样实现

中后台项目会不断出现跨页面小逻辑：时间格式化、状态映射、复制、提示、权限判断。如果不集中，维护时会找不到“全局规则在哪里”。这个项目把它们集中到 utils，是一种降低重复和保证一致性的工程选择。

## 如果重构会怎么做

- 把 `utils/permission` 提升为 `src/permission-core` 或保留但补文档。
- 区分 `utils`、`services`、`domain`：纯工具、基础服务、业务规则分别放。
- 对纯函数补单元测试，尤其是权限和数组处理。
- 为消息提示类工具增加“业务是否静默”的配置，而不是到处特判。

## 源码阅读路线

1. 先读 `src/utils/common.ts`、`array.ts`，建立纯工具概念。
2. 再读 `src/utils/message.ts`，理解全局反馈。
3. 读 `src/utils/permission/index.ts`，这是最重要的业务规则工具。
4. 读 `src/utils/theme.ts`，理解主题和平台设置。
5. 读 `src/utils/dynamics-api/*`，了解动态 API 思路。
