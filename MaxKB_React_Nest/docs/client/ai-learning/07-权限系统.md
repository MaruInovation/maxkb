# 07-权限系统

## 作用

权限系统决定“谁能看什么、谁能做什么、不同版本能开放什么能力”。这个项目权限不是单一字段，而是组合模型：

- 角色 Role：例如 ADMIN、WORKSPACE_MANAGE、USER。
- 权限 Permission：例如 APPLICATION:READ、WORKSPACE:READ+EDIT。
- 版本 Edition：例如 X-PACK-CE、X-PACK-PE、X-PACK-EE。
- 复杂权限 ComplexPermission：把角色、权限、版本按 AND/OR 组合。

关键文件：

- `src/utils/permission/type.ts`
- `src/utils/permission/data.ts`
- `src/utils/permission/index.ts`
- `src/directives/hasPermission.ts`
- `src/permission/*`
- `src/router/modules/*.ts`

## 设计思想

权限判断最终都收口到 `hasPermission()`。路由 meta、按钮指令、业务页面都不应该自己解析权限字符串。

为什么这样设计：

- 权限规则会变，调用方不应该知道所有细节。
- 企业版/专业版/社区版能力不同，需要把版本也纳入权限。
- 工作空间权限需要根据当前 workspace 动态判断。

## 权限数据从哪里来

`src/stores/modules/user.ts` 从用户资料里读取：

- `userInfo.permissions`
- `userInfo.role`
- `edition`
- `license_is_valid`

然后提供：

- `getPermissions()`
- `getRole()`
- `getEdition()`
- `isCE/isPE/isEE`

注意 `getPermissions()` 会在 PE/EE 时额外加入 `X-PACK-PE` 或 `X-PACK-EE`，这让版本能力可以像权限一样参与判断。

## 页面权限

路由文件里通过 `meta.permission` 声明访问条件。例如 `src/router/modules/application.ts` 对应用首页要求用户角色或应用读取权限。

系统路由 `src/router/modules/system.ts` 更复杂，大量使用 `ComplexPermission`，比如工作空间、角色、资源管理、共享资源、系统设置都受版本和角色影响。

## 按钮权限

`src/directives/hasPermission.ts` 注册 `v-hasPermission`。它在 created 和 beforeUpdate 时调用 `hasPermission()`，没权限就把元素 `display: none`。

为什么按钮也要权限：路由只能控制页面入口，页面内的新增、编辑、删除、导出等操作还要更细粒度控制。

## 优点

- 权限规则集中，调用方简单。
- 支持角色、权限、版本三种维度。
- 支持路由和组件两个层级。
- `ComplexPermission` 能表达复杂企业权限场景。

## 缺点

- 权限常量非常大，新人阅读压力高。
- 权限字符串和后端强耦合，缺少生成或校验机制时容易不一致。
- `v-hasPermission` 只是隐藏按钮，不等于安全，真正安全仍要靠后端鉴权。
- `display:none` 会保留 DOM 节点，某些场景可能希望直接不渲染。

## 为什么这样实现

这是典型企业级系统权限设计：前端负责体验层面的可见性和导航，后端负责真正安全。前端权限越集中，越容易保证菜单、路由、按钮三者一致。

## 如果重构会怎么做

- 从后端接口或 schema 生成权限常量，减少手写错误。
- 给 `hasPermission()` 补测试，覆盖 Role、Permission、Edition、ComplexPermission。
- 提供 `<Permission>` 组件，某些场景用条件渲染替代 `display:none`。
- 给权限文档增加业务表：每个模块需要哪些权限。
