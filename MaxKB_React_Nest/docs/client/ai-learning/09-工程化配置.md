# 09-工程化配置

## 作用

工程化配置决定项目如何开发、构建、代理、类型检查和格式化。关键文件：

- `package.json`
- `vite.config.ts`
- `tsconfig*.json`
- `eslint.config.ts`
- `.prettierrc.json`
- `env/*`
- `admin.html`、`chat.html`

## Vite 配置

`vite.config.ts` 做了几件重要事情：

- `envDir = './env'`：环境变量集中放在 env 目录。
- `createHtmlPlugin({ template: ENV.VITE_ENTRY })`：根据环境选择入口 HTML。
- `rollupOptions.input = ENV.VITE_ENTRY`：构建时指定入口。
- `outDir = dist${ENV.VITE_BASE_PATH}`：按 base path 输出。
- `renameHtmlPlugin`：构建后把入口 HTML 重命名为 `index.html`。
- `server.proxy`：代理 `/admin/api`、`/chat/api`、`/doc`、`/schema`、静态文件等。

为什么这样设计：这个项目需要支持管理端和聊天端两个入口，还可能部署在非根路径。Vite 配置不是只为本地开发服务，而是和部署形态绑定。

## package scripts

`package.json` 里有：

- `dev`: 管理端开发。
- `chat`: 聊天端开发，使用 `--mode chat`。
- `build`: 类型检查 + 管理端构建。
- `build-chat`: 类型检查 + 聊天端构建。
- `type-check`: `vue-tsc --build`。
- `lint`: ESLint 自动修复。
- `format`: Prettier 格式化 src。

这体现了基本工程纪律：构建前先类型检查。

## 双入口设计

入口文件：

- `admin.html` -> `src/main.ts`
- `chat.html` -> `src/chat.ts`

它们共享组件、样式、i18n、指令、Element Plus，但使用不同 router。这个设计适合“同一套代码仓库，不同产品入口”的场景。

## 代理设计

开发服务器代理多个路径：

- `/admin/api`
- `/chat/api`
- `/doc`
- `/schema`
- `/static`
- OSS 文件相关路径

为什么重要：前端开发不能只看页面，还要理解本地请求如何转发到后端。代理配置错了，页面代码没问题也会请求失败。

## 优点

- 多入口、多环境、代理、构建输出都集中在 Vite。
- 类型检查、构建、格式化脚本完整。
- 支持部署前缀，适合私有化部署或子路径部署。
- 使用 alias `@` 指向 `src`，减少相对路径复杂度。

## 缺点

- `main.ts` 和 `chat.ts` 初始化重复较多。
- 自定义 `renameHtmlPlugin` 使用文件系统操作，需注意跨平台和构建异常。
- `strictPort: true` 开发端口被占用时不会自动换端口。
- 没看到包体积分析脚本，性能排查时不够方便。

## 如果重构会怎么做

- 抽取 `src/setupApp.ts`，复用插件初始化。
- 把 proxy 构建逻辑拆成函数，降低 `vite.config.ts` 复杂度。
- 增加 `build:analyze` 脚本分析 bundle。
- 为环境变量写文档，说明 `VITE_ENTRY`、`VITE_BASE_PATH`、`VITE_APP_PORT` 的关系。

## 源码阅读路线

1. 先读 `package.json`，知道项目怎么跑。
2. 再读 `vite.config.ts`，理解入口、代理、输出。
3. 读 `admin.html` 和 `chat.html`，确认 HTML 到 TS 入口。
4. 读 `src/main.ts` 和 `src/chat.ts`，理解运行时注册。
5. 最后读 `env/*`，理解不同模式的环境变量。
