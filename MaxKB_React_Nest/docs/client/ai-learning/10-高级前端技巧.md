# 10-高级前端技巧

## 1. 当前项目有哪些高级设计

这个项目值得学习的高级设计不是某个语法，而是工程分层：

- 双入口：`src/main.ts` 和 `src/chat.ts` 区分管理端与聊天端。
- 模块化路由：`src/router/modules/*` 配合 `import.meta.glob`。
- 多维权限：Role、Permission、Edition、ComplexPermission 组合。
- 请求基础设施：axios 拦截器、fetch 流式请求、blob 下载、WebSocket。
- 动态表单：`src/components/dynamics-form`。
- 工作流节点协议：`src/workflow/common/data.ts` 和 `src/workflow/nodes/*`。
- 虚拟树：`src/components/folder-virtualized-tree/VirtualizedTree.vue`。
- Markdown 安全渲染：md-editor-v3 + XSS 白名单。

## 2. 哪些代码体现工程经验

`src/router/common.ts` 的 `getPermissionRoute()` 体现的是产品体验经验：用户没权限访问当前页时，不是粗暴报错，而是尝试找到可访问的同级页面。

`src/api/application/application.ts` 的动态 workspace prefix 体现的是业务上下文抽象：页面不需要知道 workspace_id 拼接规则。

`src/request/index.ts` 的统一拦截器体现的是基础设施意识：token、语言、错误、loading 不应该散落页面。

`src/components/folder-virtualized-tree/VirtualizedTree.vue` 体现性能意识：树结构数据量大时，虚拟化是架构级优化，不是微调 CSS。

## 3. 初级前端不容易看出来的点

- 路由 meta 不只是标题，它承载菜单、权限、激活状态、分组和排序。
- token 不是登录的全部，用户画像、权限、版本、工作空间才是完整上下文。
- 聊天端和管理端虽然都叫 token，但认证模型不同。
- 前端权限只改善体验，不能替代后端鉴权。
- 动态表单和工作流的价值是“让变化变成配置”，不是炫技。
- Markdown 渲染必须考虑 XSS，否则 AI 输出和用户内容会成为安全入口。

## 4. 哪些设计是为了性能

- 页面组件动态 import：降低初始加载体积。
- 双入口：减少聊天端和管理端互相拖累。
- 虚拟树：降低大目录渲染成本。
- fetch 流式请求：适合 AI 对话这类持续输出场景。
- WebSocket 封装：为实时状态或任务更新留接口。

## 5. 哪些设计是为了可维护性

- `src/api` 按业务域拆分。
- `src/stores/modules` 按状态域拆分。
- `src/router/modules` 按页面域拆分。
- `src/components` 沉淀业务组件。
- `src/permission` 和 `src/utils/permission` 集中权限规则。
- `src/locales` 集中多语言文案。

可维护性的核心不是文件多，而是“每类变化有固定位置”。新增权限去 permission，新增接口去 api，新增页面去 router 和 views。

## 6. 哪些设计是为了扩展性

- `import.meta.glob` 自动收集路由、语言、指令、动态表单组件。
- 权限常量用对象和类表达，能持续扩展。
- 工作流节点按 `type` 和 `properties` 扩展。
- 多入口和环境变量支持不同部署形态。
- 组件注册机制让公共组件扩展成本低。

## 高级前端思维总结

高级前端不是把页面写复杂，而是把复杂业务变成稳定结构。这个项目最值得你训练的能力是：

- 识别业务域边界。
- 把通用能力沉到基础设施层。
- 用路由、状态、权限、API 建立一致规则。
- 在性能问题出现前，为大数据、大组件、流式输出预留架构。
- 看到代码时能问“为什么放这里，为什么这样分层，未来变化会影响哪里”。
