# 02-状态管理

## 作用

状态管理负责保存跨页面共享的数据。这个项目使用 Pinia，把状态分成多个模块：

- `login`：管理端 token 和登录动作。
- `user`：用户信息、角色、权限、工作空间、主题、版本。
- `chat-user`：聊天端 accessToken、聊天 token、应用资料、聊天用户资料。
- `theme`：主题设置。
- `folder`、`knowledge`、`model`、`tool`、`application`：业务上下文。

入口文件是 `src/stores/index.ts`，它把所有 store 聚合成 `useStore()`。

## 设计思想

这个项目没有把所有状态堆进一个大 store，而是按业务上下文拆分。比如管理端用户用 `user`，聊天端用户用 `chat-user`，这是正确的边界划分。

为什么这样设计：

- 管理端用户和聊天端用户身份不同，token 来源和生命周期也不同。
- 权限、工作空间、主题属于“全局上下文”，放在 store 中合理。
- 页面局部状态不应该污染全局 store。

## 核心流程

管理端登录：

```text
login.asyncLogin()
-> LoginApi.login()
-> 保存 token
-> user.profile()
-> UserApi.getUserProfile()
-> 修正 workspace_id
-> 写入语言
-> theme.setTheme()
-> asyncGetProfile()
-> 写入 edition、license、version、rsaKey
```

聊天端认证：

```text
chatUser.setAccessToken()
-> getChatProfile()
-> isAuthentication()
-> setToken() 或 anonymousAuthentication()
-> applicationProfile()
-> getChatUserProfile()
```

## 优点

- `user` store 把权限、角色、版本、工作空间统一封装，路由和请求层可以复用。
- `chat-user` store 独立，避免聊天端污染后台登录态。
- `getWorkspaceId()` 做了默认值兜底，能降低接口路径出错概率。
- `useStore()` 统一引入，对业务开发者友好。

## 缺点

- `useStore()` 每次返回所有 store，类型和依赖边界不够显式。
- 部分状态同时存在内存和 localStorage/sessionStorage，需要更清楚的过期策略。
- `login.logout()` 只删除 localStorage token，没有主动清空 `this.token` 和 `user.userInfo`，页面依赖路由跳转兜底。
- `application` store 目前只有一个 `location`，它是否值得成为 store 可以重新评估。

## 为什么这样实现

大型后台系统最怕“上下文散落”。比如当前工作空间如果由每个页面自己读写，很容易出现页面 A 切换了工作空间，页面 B 还在使用旧 ID。把它收进 `user` store 后，API 层可以统一读取当前工作空间。

## 如果重构会怎么做

- 页面里优先直接导入具体 store，例如 `useUserStore()`，减少隐式依赖。
- 增加 `logoutCleanup()`，退出时清理 token、userInfo、workspace、权限缓存。
- 把 token 存储封装成独立工具，统一 localStorage/sessionStorage key。
- 为 `user.profile()` 增加错误分支说明，避免初始化失败时 UI 状态不明确。

## 源码阅读路线

1. `src/stores/index.ts`：先看 store 总入口。
2. `src/stores/modules/login.ts`：理解登录后如何建立用户态。
3. `src/stores/modules/user.ts`：重点看权限、工作空间、版本判断。
4. `src/stores/modules/chat-user.ts`：对比聊天端身份模型。
5. `src/api/application/application.ts`：看 API 如何反向依赖 store 中的 workspace。
