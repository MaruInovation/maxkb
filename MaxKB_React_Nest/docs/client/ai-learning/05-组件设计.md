# 05-组件设计

## 作用

组件体系是这个项目可维护性的核心。它不只是 Element Plus 的二次使用，而是沉淀了一批业务组件：

- `src/components/ai-chat`：聊天核心组件。
- `src/components/dynamics-form`：动态表单系统。
- `src/components/folder-tree`、`folder-virtualized-tree`：资源目录树。
- `src/components/app-table`、`app-table-infinite-scroll`：表格能力。
- `src/components/markdown`：Markdown 编辑和渲染。
- `src/components/app-icon`：业务图标体系。
- `src/workflow/common`、`src/workflow/nodes`：工作流节点组件。

## 全局组件注册

`src/components/index.ts` 把常用组件注册成全局组件。这样业务页面可以直接使用 `AppTable`、`MdPreview`、`AiChat` 等组件。

为什么这样设计：中后台项目页面很多，反复 import 基础业务组件会产生噪音。全局注册能降低业务页面成本。

代价是：组件来源不够显式，新人看到模板里的组件名时，需要知道它可能来自全局注册。

## 动态表单设计

`src/components/dynamics-form/index.ts` 用 `import.meta.glob` 自动注册动态表单下的组件。它说明项目有“表单配置驱动 UI”的需求。

这种设计适合模型参数、工具参数、认证配置这类字段类型多、变化频繁的场景。与其每个页面手写表单，不如把 TextInput、Select、Switch、Model、Knowledge 等字段变成可组合组件。

## 虚拟树设计

`src/components/folder-virtualized-tree/VirtualizedTree.vue` 使用 `@he-tree/vue` 的 virtualization 和拖拽能力。它还实现了：

- 当前节点自动展开。
- 搜索时展开匹配路径。
- 拖拽后把底层 dragContext 转成业务更容易理解的 `node-drop` 事件。

为什么这是高级点：树形目录一旦数据量大，普通递归渲染会卡。虚拟列表只渲染可视区域，能明显改善性能。

## Markdown 与安全

`src/main.ts` 和 `src/chat.ts` 配置了 `md-editor-v3`，接入 highlight、screenfull、katex、cropper、mermaid，同时用 XSSPlugin 和 `xss` 白名单控制 HTML。

这说明项目既要让 AI/用户内容富文本展示，又必须处理安全风险。允许 iframe、video、input 等标签时尤其要小心。

## 优点

- 业务组件沉淀比较多，复用意识强。
- 动态表单、Markdown、AI Chat、工作流都被做成复杂组件，而不是散落页面。
- 虚拟树、无限滚动等组件体现了性能意识。
- 图标组件和业务图标分层，能统一视觉表达。

## 缺点

- 全局注册过多会隐藏依赖。
- 某些复杂组件目录很深，新人需要阅读路线。
- 动态表单自动注册方便，但组件命名冲突需要约束。
- AI Chat 和 Workflow 都是复杂域，缺少学习文档时上手成本高。

## 如果重构会怎么做

- 给全局组件分级：基础全局、业务显式 import。
- 给 `dynamics-form` 增加字段 schema 类型文档。
- 给 `ai-chat` 拆出消息渲染、输入控制、历史、操作按钮等架构图。
- 给工作流节点建立“节点协议”文档：type、properties、config、fields。

## 源码阅读路线

1. `src/components/index.ts`：认识组件资产。
2. `src/components/dynamics-form/index.ts`：学习自动注册思想。
3. `src/components/folder-virtualized-tree/VirtualizedTree.vue`：学习性能组件。
4. `src/components/ai-chat/index.vue`：理解聊天主组件。
5. `src/workflow/common/data.ts` + `src/workflow/nodes/*`：理解工作流节点协议。
