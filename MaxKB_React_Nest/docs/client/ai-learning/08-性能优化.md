# 08-性能优化

这份文档不是泛泛讲性能，而是结合当前项目说明：哪些地方已经有性能意识，哪些地方可能是风险点。

## 1. 首屏性能

项目使用 Vite 和路由动态 import。`src/router/routes.ts`、`src/router/modules/*` 中页面组件基本都是 `component: () => import(...)`，这意味着页面级代码会按路由懒加载。

风险点在入口：`src/main.ts` 和 `src/chat.ts` 启动时同步引入了 Element Plus、所有 Element Plus icons、md-editor-v3、katex、cropper、mermaid、highlight.js 等。这些能力很强，但对首屏包体积有压力。

优化建议：

- Element Plus icons 不要全量注册，改为按需注册或局部 import。
- Markdown 编辑能力在不需要编辑器的页面按需加载。
- 聊天端如果只预览 Markdown，不一定要加载完整编辑器能力。

## 2. 重渲染问题

Pinia store 中的全局状态一旦被多个组件使用，更新会触发相关组件重新计算。`user.profile()` 会更新用户、工作空间、语言、主题、版本等多个状态，初始化阶段可以接受，但运行中频繁调用就会影响页面。

优化建议：

- 避免在页面反复调用 `user.profile()`。
- 对大型列表页面，查询条件、loading、弹窗状态优先放局部组件。
- 对复杂 computed 做输入拆分，避免依赖整个对象。

## 3. 大组件问题

风险高的区域：

- `src/components/ai-chat`
- `src/workflow`
- `src/views/application-workflow`
- `src/views/document`
- `src/views/tool`

这些模块天然复杂：聊天消息、流式输出、工作流节点、表单配置、文件导入都容易变成大组件。

优化建议：

- 按“状态逻辑、渲染组件、操作按钮、弹窗”拆分。
- 把稳定子组件做成独立组件，减少父组件更新影响。
- 对工作流节点定义统一节点协议，减少每个节点重复逻辑。

## 4. bundle 体积

潜在体积来源：

- `mermaid`
- `katex`
- `highlight.js`
- `md-editor-v3`
- `pdfjs-dist`
- `echarts`
- `LogicFlow`
- `Element Plus icons`

优化建议：

- 用 `vite build --mode chat` 后配合 rollup visualizer 分析包体积。
- 对图表、PDF、工作流、Markdown 编辑器做路由级或组件级动态 import。
- 聊天端和管理端分别检查依赖，避免聊天端携带管理端重型能力。

## 5. 懒加载

项目已经做了页面级懒加载，这是优点。还可以继续做组件级懒加载：

- 工作流编辑器只在进入工作流页面加载。
- Markdown 编辑器只在编辑场景加载。
- ECharts 只在统计图表出现时加载。
- PDF 解析只在文档预览或导入时加载。

## 6. 缓存设计

已有缓存：

- 管理端 token 存 localStorage。
- 聊天端 token 同时存 sessionStorage/localStorage，key 带 accessToken。
- locale 存 localStorage。
- workspace_id 存 localStorage。

风险：

- localStorage 长期存在，退出登录或切换身份时需要清理完整。
- 用户资料没有明显缓存失效策略，依赖路由初始化拉取。

优化建议：

- 封装 storage key，统一清理策略。
- 用户资料增加“已加载时间”或显式刷新入口。
- 聊天端 token 是否需要长期 localStorage，要根据产品安全要求决定。

## 7. 请求优化

`src/request/index.ts` 已经统一了 loading、错误、token、语言，这是请求优化的基础。

可优化点：

- 对重复请求增加取消机制，例如搜索、过滤、自动补全。
- 对列表查询增加防抖。
- 对大文件下载和导入提供进度反馈。
- 对 401/403/404 的处理策略配置化，减少 URL 特判。

## 8. 图片优化

项目包含大量 SVG 图标和少量 PNG，例如 `src/assets/workflow-demo.png`。SVG 适合图标，PNG 适合真实截图或演示图。

优化建议：

- 大 PNG 用压缩工具处理。
- 业务图标保持 SVG，避免位图放大模糊。
- 对用户上传图片预览使用懒加载和对象 URL 回收。

## 9. 长列表优化

项目已经在树组件上使用虚拟化：`src/components/folder-virtualized-tree/VirtualizedTree.vue` 使用 `@he-tree/vue` 的 virtualization。

表格方面有：

- `src/components/app-table-infinite-scroll`
- `el-table-infinite-scroll`
- `src/components/infinite-scroll`

优化建议：

- 超大表格不要只靠分页，也要关注行渲染成本。
- 文件树、资源树、聊天历史适合虚拟列表。
- 拖拽树要避免每次拖拽都深拷贝整棵树。

## 10. Vue 性能问题

本项目是 Vue，不是 React。重点关注：

- 大对象响应式：不要把后端大对象无脑放入深层 reactive。
- v-for key：长列表必须稳定 key。
- computed 依赖：避免 computed 依赖整个 store。
- 全局组件注册：启动成本和可发现性之间要权衡。
- 条件渲染：权限隐藏用 `display:none` 不销毁组件，复杂组件可考虑 `v-if`。

## 最值得学习的性能思想

性能不是最后才优化。这个项目已经在架构层做了几件正确的事：

- 路由懒加载。
- 管理端和聊天端双入口。
- 请求层统一封装，方便做全局优化。
- 虚拟树处理大数据目录。
- Markdown 使用 XSS 白名单，避免为了功能牺牲安全。

你学习性能时，要先学会找“成本来源”：启动成本、渲染成本、网络成本、状态更新成本、资源体积成本。不要一上来就背优化清单。
