# src/router/index.ts

## NProgress

页面跳转进度条库，它用在路由切换时，给用户一个“页面正在加载”的顶部进度条反馈

## beforeEach

`set_next_route(to)` 是项目自己封装的小工具，用来记录“当前准备进入的路由”。

流程：

1. 404 直接放行，不做登录和权限判断。

2. 判断当前路由是否在 `notAuthRouteNameList` 白名单中。

3. 如果不在白名单：

   - 如果 query 中有 token，写入 localStorage。
   - 读取 token。
   - 没有 token，跳转 `/login`。
   - 有 token 但没有用户信息，调用 `user.profile()` 拉取用户上下文。

4. 调用 `set_next_route(to)`，保存目标路由。

5. 判断 `to.meta.permission`：

   - 没有配置权限，默认放行。
   - 配置了权限，调用 `hasPermission(..., 'OR')` 判断。

6. 如果没有权限：
   - 调用 `getPermissionRoute(routes, to)` 找一个当前用户能访问的路由。
   - 找不到则进入 `noPermission`。

# src/router/common.ts

这个文件最核心的工程思想是：

    权限失败时，不要只做“禁止访问”，还要尽量帮用户找到一个可访问的页面。

```
路由守卫发现没权限
-> getPermissionRoute(routes, to)
-> getSameRouteList 找同级页面
-> hasPermission 过滤
-> findAccessibleRoute 找可进入子页面
-> 返回一个可跳转 route
```

## getChildRouteList

作用：根据 path + name 找到某个路由的 children。

比如布局、菜单、面包屑想知道某个路由下面有哪些子页面，就可以用它。

```
用在:

getChildRouteListByPathAndName(...)
```

## getSameRouteList

作用：根据路由 name 找到它所在的同级路由数组。

```
举个例子：

/system/setting/theme
/system/authentication
/system/email

这几个如果在同一个 children 里，那么找到其中一个，就能拿到这一组同级路由
```

## getPermissionRoute

作用：当前用户没权限访问 to 时，找一个可以替代进入的路由。

```
大致逻辑：

1. 先确定 routeName
   - 如果 to.meta.sameRoute 存在，用 sameRoute
   - 否则用 to.name

2. 找到 routeName 所在的同级路由列表

3. 在同级路由里找第一个满足权限的 route

4. 如果这个 route 有 children
   - 继续找 children 里第一个可访问的叶子路由

5. 如果找到了，并且不是当前路由
   - 返回 { name: finalRoute.name, params: to.params }

6. 如果同级找不到
   - 从全局 routes 里找第一个可访问路由

7. 仍然找不到
   - 返回 { name: 'noPermission' }

```

## findAccessibleRoute(routes)

作用：递归找第一条当前用户有权限访问的路由。

它会跳过两类路由：

```ts
// 没权限的路由跳过
if (permission && !hasPermission(permission as any, 'OR')) {
  continue
}

// 动态参数路由跳过
if (route.path.includes(':')) {
  continue
}
```

# 简单路由板块

    已 src/router/modules/application.ts 为例

meta 是路由系统和菜单系统、权限系统之间的连接层。router 不只是页面跳转工具，它也是中后台应用的导航配置中心

```ts
meta: {
  title,       // 页面/菜单标题，通常是 i18n key，比如 'views.application.title'
  menu,        // 是否作为主菜单显示
  permission,  // 进入该路由需要的权限配置
  icon,        // 菜单默认图标
  iconActive,  // 菜单激活图标，有些路由会配置
  group,       // 菜单分组，比如 workspace
  order,       // 菜单排序
}

```

tip：child里面的routePath前面加上'/'说明直接跳转到

# 复杂路由板块

    已 src/router/modules/system.ts 为例

## ComplexPermission

    复合权限表达式

```
这个项目权限是三维的
角色维度：ADMIN / WORKSPACE_MANAGE / USER
权限维度：APPLICATION:READ、USER_MANAGEMENT:READ 等
版本维度：CE / PE / EE
```

### class ComplexPermission

参数:

```
如果 compare 是 AND：
必须同时满足权限、角色、版本。
如果 compare 是 OR：
满足权限或角色其中一个，并且版本满足。


roleList        允许哪些角色
permissionList  允许哪些具体权限
editionList     允许哪些版本
compare         角色和权限之间怎么组合，OR 或 AND
```

真正的判断在 src/utils/permission/index.ts：

```ts
/**
如果 compare 是 AND：
必须同时满足权限、角色、版本。
如果 compare 是 OR：
满足权限或角色其中一个，并且版本满足。
 */
const permissionOk = permission.permissionList.some(...)
const roleOk = roleList.some(...)
// 不传版本限制，就代表所有版本都可以。
const editionOK =
  permission.editionList.length > 0
    ? editionList.some((e) => edition.toString() == e.toString())
    : true

return permission.compare === 'AND'
  ? permissionOk && roleOk && editionOK
  : (permissionOk || roleOk) && editionOK

```

editionList 为空时，editionOK 默认 true。举个 system.ts 里的例子：

```ts
/**
 *
  必须是 EE 版本；
并且满足下面任意一个：
1. 是 ADMIN
2. 有 RESOURCE_APPLICATION_READ 权限
 */

new ComplexPermission(
  [RoleConst.ADMIN],
  [PermissionConst.RESOURCE_APPLICATION_READ],
  [EditionConst.IS_EE],
  'OR',
)
```
