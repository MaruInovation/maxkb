# 01-路由系统

## 作用

路由系统负责三件事：

- 把 URL 映射到页面组件。
- 根据登录状态决定是否能进入页面。
- 根据权限决定进入当前页、替代页还是无权限页。

关键文件：

- `src/router/index.ts`：管理端路由实例和守卫。
- `src/router/routes.ts`：管理端路由聚合。
- `src/router/modules/*.ts`：业务模块路由。
- `src/router/chat/index.ts`：聊天端路由守卫。
- `src/router/common.ts`：查找子路由、同级路由、可访问路由。

## 设计思想

`src/router/routes.ts` 使用 `import.meta.glob('./modules/*.ts', { eager: true })` 自动收集模块路由。这是典型的大型前端做法：每个业务模块维护自己的路由文件，总路由只负责聚合。

为什么这样设计：

- 应用、知识库、工具、系统等模块可以独立维护。
- 新增模块时不用改一个巨大的路由数组。
- 路由 meta 可以承载菜单、图标、排序、权限、激活菜单等导航信息。

## 管理端守卫流程

`src/router/index.ts` 的流程：

```text
beforeEach
-> NProgress.start()
-> 404 直接放行
-> 非白名单路由检查 token
-> 无 token 跳 login
-> 无 userInfo 则 user.profile()
-> set_next_route(to)
-> hasPermission(to.meta.permission)
-> 有权限 next()
-> 无权限 getPermissionRoute(routes, to)
```

这里的高级点是 `getPermissionRoute`：它不是简单跳无权限页，而是尝试在同级路由里找一个用户有权限访问的页面。对中后台来说，这能减少“进模块后第一个 tab 无权限”的尴尬体验。

## 聊天端守卫流程

`src/router/chat/index.ts` 面向公开聊天入口，逻辑和管理端不同：

```text
读取路由 accessToken
-> chatUser.setAccessToken()
-> 判断应用是否开启认证
-> 如果需要认证且没有 token，跳聊天登录页
-> 如果不需要认证，走 anonymousAuthentication()
-> 加载 applicationProfile 和 chatUserProfile
```

为什么聊天端单独写守卫：聊天入口不是后台用户访问，而是应用分享访问。它要处理匿名、密码、LDAP、钉钉、企业微信、飞书等登录方式，不能复用管理端用户登录模型。

## 优点

- 模块路由拆分清晰，适合大型业务扩展。
- 路由 meta 同时服务菜单和权限，信息集中。
- 路由守卫把登录校验前置，页面组件可以少写重复判断。
- 路由组件使用动态 import，有天然懒加载效果。

## 缺点

- `routes.ts` 中删除 `model.ts` 的逻辑比较隐蔽，新人容易困惑。
- 路由 meta 字段很多，缺少统一类型约束，后期容易拼错。
- 管理端和聊天端的入口代码有大量重复初始化逻辑。
- `router.push('/404 ')` 里路径有尾随空格，虽然不是本次修改目标，但这是值得排查的细节。

## 如果重构会怎么做

- 为 `meta` 定义统一 TypeScript 类型，例如 `AppRouteMeta`。
- 把 `main.ts` 和 `chat.ts` 重复的插件注册抽成 `setupApp(app, options)`。
- 为 `getPermissionRoute` 增加单元测试，覆盖无权限、动态路由、同级 fallback。
- 明确模型路由为什么被排除，避免 `delete modules['./modules/model.ts']` 这种隐式规则。

## 源码阅读路线

1. 先读 `src/router/routes.ts`，理解总路由如何聚合。
2. 再读 `src/router/index.ts`，理解登录和权限拦截。
3. 然后读 `src/router/common.ts`，理解无权限 fallback。
4. 最后挑 `src/router/modules/application.ts` 和 `src/router/modules/system.ts` 对比：前者简单，后者复杂，最能体现权限路由设计。
