# 00-项目总览

这份文档先帮你建立“项目地图”。看大型前端项目时，不要从某个页面组件开始钻细节，而要先找到入口、路由、状态、请求、权限、组件体系这几条主线。这个项目的主线很清楚：它是一个基于 Vue 3 的 MaxKB 前端，既有管理端，也有聊天端。

## 1. 项目整体架构

核心结构可以这样理解：

```text
admin.html -> src/main.ts -> src/router/index.ts -> src/router/routes.ts -> src/views/*
chat.html  -> src/chat.ts -> src/router/chat/index.ts -> src/router/chat/routes.ts -> src/views/chat/*

src/request/*       统一请求层
src/stores/*        Pinia 状态层
src/api/*           业务 API 层
src/permission/*    业务权限模型
src/utils/*         工具与跨模块能力
src/components/*    全局组件与复杂业务组件
src/workflow/*      工作流编排核心
src/locales/*       国际化
```

高级前端看项目时，第一步不是问“这个按钮怎么写”，而是问“用户的一次操作经过哪些层”。在本项目里，一次管理端操作通常经过：页面组件 -> API 模块 -> `src/request/index.ts` -> 后端 -> Pinia 或局部状态 -> 组件重新渲染。

## 2. 技术栈分析

项目技术栈来自 `package.json`：

- Vue 3.5 + Composition API：适合大型中后台拆分组件和逻辑。
- Vite 6：负责多入口构建、开发服务器、代理和构建输出。
- Pinia 3：管理用户、登录、主题、知识库、应用、聊天用户等全局状态。
- Vue Router 4：管理管理端和聊天端两套路由。
- Element Plus：中后台基础 UI 组件库。
- vue-i18n：多语言能力。
- axios + fetch：普通请求用 axios，流式请求用 fetch。
- LogicFlow、@antv/layout：支撑工作流节点编排。
- md-editor-v3、marked、highlight.js、katex、mermaid、DOMPurify/xss：支撑 Markdown 编辑、预览、代码高亮、公式、图表和安全过滤。
- @he-tree/vue：支撑虚拟树和拖拽树。

为什么这样设计：MaxKB 不是简单表单系统，它包含知识库、应用、工作流、聊天、权限、模型、工具等复杂域。技术栈选择的重点不是“新”，而是“能把复杂业务拆成可维护的层”。

## 3. 项目解决什么问题

从源码命名看，项目主要解决这些问题：

- 管理智能体应用：`src/views/application`、`src/api/application/application.ts`。
- 管理知识库和文档：`src/views/knowledge`、`src/views/document`、`src/api/knowledge`。
- 配置模型、工具、MCP、触发器：`src/views/model`、`src/views/tool`、`src/views/mcp`、`src/views/trigger`。
- 编排工作流：`src/workflow`、`src/views/application-workflow`。
- 对外提供聊天端：`src/chat.ts`、`src/router/chat`、`src/components/ai-chat`。
- 管理系统权限、角色、工作空间：`src/router/modules/system.ts`、`src/permission`、`src/utils/permission`。

新人容易只看到页面，忽略“业务域”。高级前端会先识别业务域，因为业务域决定目录、状态、权限和 API 组织方式。

## 4. 项目核心模块

| 模块     | 关键文件                                                                            | 为什么重要                                           |
| -------- | ----------------------------------------------------------------------------------- | ---------------------------------------------------- |
| 应用入口 | `src/main.ts`、`src/chat.ts`                                                        | 决定应用启动时注册哪些插件、路由、组件、指令         |
| 路由系统 | `src/router/index.ts`、`src/router/routes.ts`、`src/router/modules/*`               | 决定页面组织、权限跳转、懒加载边界                   |
| 状态管理 | `src/stores/index.ts`、`src/stores/modules/*`                                       | 把跨页面共享数据从组件里抽出来                       |
| 请求封装 | `src/request/index.ts`、`src/request/chat/index.ts`                                 | 统一 token、语言、错误处理、loading、下载、WebSocket |
| 权限系统 | `src/utils/permission/*`、`src/permission/*`、`src/directives/hasPermission.ts`     | 同时控制路由访问和按钮显示                           |
| 组件体系 | `src/components/index.ts`、`src/components/dynamics-form`、`src/components/ai-chat` | 把高频 UI 和复杂业务组件沉淀为复用资产               |
| 工作流   | `src/workflow/common/data.ts`、`src/workflow/nodes/*`                               | 项目复杂度最高的图形化编排域                         |
| 工程配置 | `vite.config.ts`、`env/*`、`package.json`                                           | 支撑双入口、多环境、代理和构建输出                   |

## 5. 页面运行流程

管理端运行流程：

```text
浏览器打开 admin.html
-> 加载 src/main.ts
-> 注册 Pinia、Element Plus、全局组件、全局指令、i18n、router
-> RouterView 渲染 src/App.vue
-> router.beforeEach 检查 token 和用户信息
-> 根据 routes + meta.permission 选择可访问页面
-> 页面组件调用 api/*
-> request/index.ts 注入 AUTHORIZATION 和 Accept-Language
```

聊天端运行流程：

```text
浏览器打开 chat.html
-> 加载 src/chat.ts
-> 使用 src/router/chat/index.ts
-> 从路由参数 accessToken 定位具体应用
-> chatUser store 判断是否需要认证
-> 匿名认证、密码认证或第三方登录
-> 加载应用资料和聊天用户资料
-> 渲染 src/components/ai-chat
```

为什么拆成双入口：管理端和聊天端的路由、认证、API baseURL 都不同。拆入口可以避免聊天端加载过多管理端页面逻辑，也让构建产物更贴近部署形态。

## 6. 数据流

项目的数据流大体是单向的：

```text
用户操作
-> Vue 组件事件
-> api 模块方法
-> request 封装
-> 后端接口
-> 返回 Result
-> 写入 Pinia 或组件局部 ref
-> 模板重新渲染
```

例子：应用列表来自 `src/api/application/application.ts`。该文件用一个动态 `prefix.value` 拼出 `/workspace/{workspace_id}/application`，而 `workspace_id` 从 `user` store 读取。这个设计把“当前工作空间”从页面里移出去，页面只关心业务动作，不关心 URL 细节。

## 7. 登录流程

管理端登录流程在 `src/stores/modules/login.ts`：

```text
登录表单提交
-> LoginApi.login / ldapLogin / 第三方 callback
-> 拿到 token
-> localStorage.setItem('token')
-> user.profile()
-> 拉取用户、工作空间、语言、主题、版本、许可证信息
```

路由守卫在 `src/router/index.ts` 负责兜底：访问非白名单路由时，如果没有 token，跳转 `/login`；如果没有 `user.userInfo`，调用 `user.profile()`。

为什么这样设计：登录不是只拿 token。大型系统登录后要立即建立“用户上下文”，包括权限、工作空间、语言、主题和版本能力，否则页面无法正确显示菜单、按钮和资源范围。

## 8. 权限流程

权限流程有四层：

```text
后端返回 userInfo.permissions / role / edition
-> src/stores/modules/user.ts 转成 getPermissions/getRole/getEdition
-> src/utils/permission/index.ts 统一判断 Role、Permission、Edition、ComplexPermission
-> 路由 meta.permission 控制页面访问
-> v-hasPermission 控制按钮或局部 UI
```

`src/router/modules/system.ts` 里大量使用 `ComplexPermission`，说明系统权限不是简单“有/无”，而是角色、权限、版本三者共同决定。

为什么这样设计：企业版、专业版、社区版能力不同；管理员、工作空间管理员、普通用户能力也不同。如果把判断散落在页面里，后期维护会非常痛苦。

## 9. API 流程

管理端 API 使用 `src/request/index.ts`：

- baseURL 是 `(window.MaxKB?.prefix || '/admin') + '/api'`。
- 请求拦截器注入 `Accept-Language` 和 `AUTHORIZATION`。
- 响应拦截器处理业务 code、401、403、404。
- `get/post/put/del` 包装 loading。
- `postStream` 用 fetch 支撑流式响应。
- `exportFile/exportExcel` 处理 blob 下载。
- `socket` 统一创建 WebSocket URL。

聊天端 API 使用 `src/request/chat/index.ts`，baseURL 改为 `/chat/api`，token 来自 `chatUser` store。为什么分开：管理端 token 和聊天端 token 的生命周期、存储 key、接口前缀不同。

## 10. 状态管理流程

`src/stores/index.ts` 把多个 Pinia store 汇总成 `useStore()`：

```ts
const { user, login, theme, application, chatUser } = useStore();
```

好处是页面引入统一，坏处是类型推导和 tree-shaking 不如直接导入具体 store 清晰。这个项目选择统一入口，说明它更看重复使用体验和团队约定。

状态模块大致分三类：

- 用户上下文：`login`、`user`、`chat-user`。
- 业务上下文：`application`、`knowledge`、`model`、`tool`、`folder`。
- UI 上下文：`common`、`theme`、`prompt`。

学习重点：不要把所有东西都放 Pinia。只有跨页面、跨组件、生命周期较长的状态才适合进入 store；表单输入、弹窗开关、表格 loading 更适合放组件局部。
