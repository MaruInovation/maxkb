# 工程化格式化与提交钩子

## 为什么放在根目录

Prettier、lint-staged 和 Husky 属于 monorepo 的工程约束，不属于某一个应用。

如果把格式化规则分别放进 client 和 server，后续很容易出现前后端风格不一致的问题。放在根目录的好处是：所有 workspace 共享一套提交前检查，开发者只需要记住一个入口。

## 工具分工

- Prettier 负责统一代码格式。
- lint-staged 只处理本次暂存的文件，避免每次提交都格式化整个仓库。
- Husky 负责把检查接入 Git pre-commit 阶段。

## 为什么不是只靠编辑器

编辑器配置只能约束当前开发者的本地环境，不能保证团队里每个人都一致。

Git hook 是提交前的最后一道工程防线。它不替代编辑器格式化，而是防止未格式化代码进入历史记录。

## 源码阅读路线

1. 先看根目录 `package.json` 的 `format`、`format:check`、`prepare` 和 `lint-staged` 配置。
2. 再看 `.prettierrc.json`，理解项目统一格式规则。
3. 最后看 `.husky/pre-commit`，理解提交前如何触发 lint-staged。
